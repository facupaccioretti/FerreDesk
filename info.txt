# Implementación del Worker Centralizado (SaaS Multi-tenant)

Se construirá un worker centralizado continuo dentro de un único comando de Django (`worker_tareas_pendientes.py`) para consumir las colas de manera eficiente, evitando costos adicionales en Render y garantizando alta resiliencia.

## User Review Required

> [!IMPORTANT]
> El enfoque acordado elimina la necesidad de One-Off Jobs o Cron Jobs pagos. Vamos a embeber un worker persistente protegido. Se requiere confirmación antes de iniciar la programación.

## Proposed Changes

### Backend (Comandos y Servicios)

#### [NEW] `ferreapps/sistema/management/commands/worker_tareas_pendientes.py`
Crearemos el núcleo del worker. Este comando se quedará vivo en un `while True:` y hará lo siguiente:
1.  **Cierre de conexiones:** Llamará a `django.db.close_old_connections()` en cada iteración para evitar el clásico error de PostgreSQL de conexión cerrada por inactividad.
2.  **Límites por ciclo:** Extraerá como máximo un bloque (ej. 5 o 10) de cargas iniciales y listas de precios, iterando sobre los tenants activos.
3.  **Recuperación de trabados:** Buscará importaciones o cargas iniciales en estado `ESTADO_PROCESANDO` que tengan más de 1 hora de antigüedad, y las devolverá a `ESTADO_ERROR` con un mensaje indicando que el proceso falló inesperadamente.
4.  **Graceful Shutdown:** Atraparemos la señal `SIGTERM` (la que envía Render al apagar o reiniciar el contenedor) para que el worker intente terminar ordenadamente la tarea actual antes de cerrarse o al menos salir limpio.

#### [MODIFY] `ferreapps/productos/services/importacion_lista_precios_service.py`
Se actualizará la consulta de obtención de tarea para hacerla segura ante concurrencia (por si Render lanza dos contenedores solapados durante un deploy):
Se ajustará el código para que al tomar la tarea se controle de manera robusta usando bloqueos condicionales si fuera necesario o asegurando que el estado actualice correctamente.

#### [MODIFY] `ferreapps/proveedores/services/carga_inicial_proveedor_service.py`
Misma protección concurrente y optimización para el flujo de carga inicial.

### Infraestructura (Render)

#### [MODIFY] `start.prod.sh` (en la raíz o carpeta de scripts)
Se ajustará el archivo de arranque de Render para incluir el escudo nivel 2 (el bucle bash) condicionado por variable de entorno:
```bash
if [ "$ENABLE_EMBEDDED_WORKER" = "true" ]; then
  (
    while true; do
      python manage.py worker_tareas_pendientes
      echo "El worker se cerró, reiniciando en 5 segundos..."
      sleep 5
    done
  ) &
fi
```

### Manual Verification
1.  Arrancar el `start.prod.sh` o el entorno local con `ENABLE_EMBEDDED_WORKER=true`.
2.  Cargar un archivo desde el frontend.
3.  Observar el output del worker en consola.
4.  Detener la base de datos a la mitad del proceso para simular un crash, y verificar que:
    *   La tarea interrumpida sea recuperada (marcada como Error) en el siguiente ciclo al volver a conectarse.
